# ADR 0002: 不创建中间文件格式

## 状态

已采纳

## 背景

讨论了是否创建一种新的文件格式（".xxx"），将语义层和视觉层数据持久化，作为所有转换的"中心格式"。历史上 XSL-FO（1999）、UOML（2008）、CDF（2010）都尝试过类似路径，均以失败告终。

## 决策

**不创建持久化的中间文件格式。** 双层模型仅作为内存对象存在，转换器通过内存对象通信。调试时用 JSON/MessagePack 做临时快照。

## 理由

1. 要成为所有源格式的超集，schema 需要覆盖 OOXML 6000+ 页规范和 PDF 1000+ 页规范的并集，维护成本不可承受。
2. 历史案例高度一致地表明这条路径失败——用户不会为"格式统一性"放弃对最终输出的精确控制。
3. 内存对象的演化没有兼容性包袱——Reader/Writer 之间是接口契约，不是文件格式契约。
4. JSON 快照满足调试需求，不需要定义独立的序列化规范。

## 后果

- 不同转换任务之间无法通过文件"断点续转"——每次都是源文件→IR→目标文件的新流程。这对 V1 是合理的，因为转换是单次操作。
- Reader/Writer 之间的接口设计成为架构核心，需在编码前明确定义。
- 如需未来支持"保存中间状态"，MessagePack 快照可以自然演进为持久化格式，但不承诺向前兼容。
